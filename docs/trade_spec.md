## 1. 交易の種類

| 種別 | 相手 | タイミング |
|------|------|-----------|
| バンク交易 | バンク（中央在庫） | 自分のターンの「交易フェーズ」のみ |
| 港交易 | バンク（港レートで） | 自分のターンの「交易フェーズ」のみ |
| プレイヤー間交易 | 他プレイヤー | 自分のターンの「交易フェーズ」のみ |

> **注**: ダイスロール後かつ建設フェーズ前が「交易フェーズ」。同ターン内で交易と建設は任意の順序・回数で行える。

---

## 2. バンク交易（4:1）

### 2-1. ルール

- **レート**: 同じ資源**4枚**をバンクに渡し、任意の資源**1枚**を受け取る。
- 港を所有していない場合のデフォルト交易レート。
- 同一資源を4枚持っている場合のみ実行可能。

### 2-2. 処理フロー

```
Player → [resource_type: X, amount: 4] → Bank
Bank  → [resource_type: Y, amount: 1] → Player

Validation:
  player.hand[X] >= 4
  bank.stock[Y] >= 1
```

## 3. 港交易（Harbor Trade）

### 3-1. 汎用港（3:1）

- 海岸辺に接する頂点に開拓地/都市を持つ場合に利用可能。
- **レート**: 任意の資源**3枚**（同種）→ 任意の資源**1枚**。

### 3-2. 特殊港（2:1）

- 該当資源の港頂点に開拓地/都市を持つ場合に利用可能。
- **レート**: 対応資源**2枚** → 任意の資源**1枚**。

| 港種別 | 必要資源 | 枚数 | 受取 |
|--------|---------|------|------|
| 木材港 | wood | 2 | 任意1 |
| レンガ港 | brick | 2 | 任意1 |
| 羊毛港 | wool | 2 | 任意1 |
| 麦港 | grain | 2 | 任意1 |
| 鉄鉱石港 | ore | 2 | 任意1 |
| 汎用港 | 任意（同種） | 3 | 任意1 |

### 3-3. 港アクセス判定

```
harbor_access(player, harbor) =
  harbor.vertices.any(v => v.owner == player.id)

effective_rate(player, resource_type):
  rates = [4]  // デフォルト
  if harbor_access(player, generic_harbor):   rates.append(3)
  if harbor_access(player, specific_harbor[resource_type]): rates.append(2)
  return min(rates)
```

> **注**: 複数の港を保有する場合、最も有利なレートが適用される。2:1 > 3:1 > 4:1の優先順位。

**港アクセスの有効タイミング（R-06）**:

- 初期配置フェーズ中に港頂点へ開拓地を置いた場合、港アクセスは**ゲーム本番開始（最初のプレイヤーのダイスロール）から有効**になる。
- 初期配置フェーズ中は交易フェーズ自体が存在しないため、初期配置中に港レートを使用することはできない。
- 実装上は `GamePhase` が `SETUP` から `MAIN` に遷移した時点で `effective_rate()` が呼び出せるようになる。

## 4. プレイヤー間交易（Domestic Trade）

### 4-1. 基本ルール

- アクティブプレイヤーのみがオファーを発起できる。
- 非アクティブプレイヤーはオファーへの応答のみ可能。
- 非アクティブプレイヤー同士の交易は**不可**。
- 資源カードのみ交易対象（発展カード・特殊カードは不可）。
- 取引は**1対1の同期的な合意**が必要（一方的な押し付け不可）。

### 4-2. オファーフロー

```
State: TRADE_OFFER
  active_player sends:
    offer  = { resource_type: X, amount: N }   // 渡す
    request= { resource_type: Y, amount: M }   // 要求
    targets= [player_id, ...]                  // 対象（全員 or 指定）

State: TRADE_RESPONSE（全対象プレイヤーが並行して応答）
  each target_player responds:
    ACCEPT | REJECT | COUNTER_OFFER

State: TRADE_CONFIRM（アクティブプレイヤーが受諾者を1人選ぶ）
  active_player selects: accepting_player_id | CANCEL

State: TRADE_EXECUTE
  swap(active_player.hand[X] -= N, target_player.hand[X] += N)
  swap(active_player.hand[Y] += M, target_player.hand[Y] -= M)
```

### 4-3. カウンターオファー

- 受け取り側は金額・種類を変更した逆オファーを送れる。
- カウンターオファーはアクティブプレイヤーが改めてACCEPT/REJECTを選択。
- カウンターオファーへのカウンターオファーは**不可**（1往復まで）。

**ACCEPT確定後・実行前の再バリデーション（必須）**:

```
pre_execute_validate(active_player, target_player, offer, request):
  // 送り手の手持ちを再確認（独占カード等で資源が変動している可能性がある）
  assert active_player.hand[offer.resource] >= offer.amount
  // 受け手の手持ちを再確認（同上）
  assert target_player.hand[request.resource] >= request.amount
  // バンク在庫は関与しないため不要
  if validation fails:
    → TRADE_CANCELLED 状態へ遷移し、両プレイヤーに通知
    → 交易フェーズに戻る（ペナルティなし）
```

> **実装注**: `TRADE_CONFIRM`（受諾者選択）から`TRADE_EXECUTE`へ遷移する直前に必ず呼び出すこと。楽観的ロックではなく、実行時点の GameState 値で判定する。

### 4-4. バリデーション

| チェック項目 | ルール |
|-------------|--------|
| 送り手の手持ち | `sender.hand[offer.resource] >= offer.amount` |
| 受け手の手持ち | `receiver.hand[request.resource] >= request.amount` |
| 自己取引 | アクティブプレイヤー自身との交易は不可 |
| ゼロ取引 | offer・request両方ともに amount > 0 であること |
| ターンチェック | 交易フェーズ中のみ実行可能 |

## 5. 交易フェーズの制約

| ルール | 詳細 |
|--------|------|
| 回数制限 | バンク・港交易は**無制限**（バンク在庫の許す限り） |
| プレイヤー間交易回数 | **無制限**（ただし現実的に合意できる回数） |
| 交易後の建設 | 可能（同フェーズ内で交互に行える） |
| 強制交易 | 不可（合意が必要） |
| 秘密交易 | 不可（全プレイヤーがオファー内容を閲覧可能） |

---

## 6. エラー・例外ケース

| ケース | 処理 |
|--------|------|
| バンク在庫が不足 | 交易を拒否、UIにエラー表示 |
| 受け手が途中で資源を失う | 確定前に再バリデーション実施（§4-3参照） |
| 全プレイヤーがREJECT | オファーをキャンセルし交易フェーズに戻る |
| タイムアウト（人間プレイヤー） | **60秒**経過で未応答はREJECT扱い。残り秒数をUIに表示する |
| タイムアウト（AIプレイヤー） | **3秒**以内に応答。3秒超過は実装バグとして警告ログを出力 |

## 7. UI要件サマリー（Fenn向け）

| 要素 | 仕様 |
|------|------|
| 交易ボタン | ダイスロール後にアクティブ化 |
| レート表示 | 現在の有効レートをリアルタイム表示（例: wool: 2:1） |
| オファーモーダル | 渡す資源・受け取る資源を選択するUI |
| 応答UI | 他プレイヤー側にACCEPT/REJECT/COUNTERボタン |
| 確定UI | 複数ACCEPTの場合、アクティブプレイヤーが相手を選択 |
| 交易ログ | 全交易内容を履歴として表示 |
